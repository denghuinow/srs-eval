# **A. 多维检查点抽取 Prompt（从 Reference SRS 自动生成检查点）——中文版**

你可以直接把 `{reference_srs}` 替换为你的真实需求文档。

---

**（可直接使用的完整 Prompt）**

```text
你是一名资深需求分析专家与需求质量工程师。

你的任务是：从下面的「参考需求文档（Reference SRS）」中，抽取 **可量化、可验证、客观、原子化** 的多维度需求检查点（Checkpoints）。

这些检查点将作为评估其他候选需求文档的唯一基准。

你必须严格按照以下五大维度进行抽取，不得遗漏，也不得跨文档推断。

-------------------------------------------------
【检查点维度（必须全部输出）】
-------------------------------------------------

1. FUNCTIONAL（功能性检查点）
   - 抽取文档中所有功能行为、操作流程、系统响应、业务规则。
   - 每个检查点必须是可验证的“单一行为”或“单一规则”。
   - 示例：系统支持用户通过邮箱密码注册。

2. BUSINESS_FLOW（业务流程检查点）
   - 抽取所有流程步骤、主流程、子流程、分支条件、前置条件、后置条件、状态迁移。
   - 每个检查点代表流程中的“一步”或“一条规则”。
   - 示例：注册成功后必须发送验证邮件。

3. BOUNDARY（边界条件检查点）
   - 抽取所有范围、最大值、最小值、长度限制、容量限制、时间限制等边界条件。
   - 示例：密码长度需在 8–20 位之间。

4. EXCEPTION（异常处理检查点）
   - 抽取所有异常流程、错误处理、失败策略、降级策略、重试逻辑等。
   - 示例：登录失败超过 5 次需锁定账户。

5. CONSISTENCY_RULE（一致性与非冲突规则检查点）
   - 抽取文档中描述的所有逻辑一致性规则，包括：
     - 权限规则
     - 状态机规则
     - 数据有效性规则
     - 互斥条件
     - 不允许发生的情况
   - 用于检测其他文档是否违反参考文档逻辑。
   - 示例：订单状态不得跳过已支付状态直接进入已完成状态。

-------------------------------------------------
【输出格式（非常严格）】
-------------------------------------------------

输出必须是 **TSV 格式**，使用 **制表符 TAB** 分隔。

表头必须为：

Category	Checkpoint

规则：
- 每行一个检查点
- 不得编号
- 不得添加解释说明
- 不得包含空行、注释
- 所有检查点必须完全来自参考文档内容
- 每个检查点必须是客观、可验证、原子、无歧义的句子

-------------------------------------------------
【参考需求文档 Reference SRS】
-------------------------------------------------

{reference_srs}

-------------------------------------------------
请开始输出（仅输出 TSV，无其他文本）
-------------------------------------------------
```

# **B. 多维核查 Prompt（对 Candidate SRS 进行 yes/no 逐条核查）——中文版**

将 `{checkpoints_table}` 替换为 A 步骤生成的 TSV 检查点列表。
将 `{candidate_srs}` 替换为你要评估的新文档。

---

**（可直接使用的完整 Prompt）**

~~~text
你是一名专业的需求文档核查引擎。

你的任务是：根据提供的检查点列表（Checkpoints），对候选文档逐条进行核查，并对每个检查点输出 yes/no（是否明确包含）。

-------------------------------------------------
【核查规则】
-------------------------------------------------

1. 必须逐条核查（从上到下，顺序不变）。
2. 判断必须完全基于候选文档内容，不得：
   - 推测
   - 衍生逻辑
   - 使用外部知识
   - 做模糊匹配
3. 判断为二元制：
   - “yes” = 候选文档中明确且无歧义地描述了该检查点
   - “no”  = 未出现、描述不清、或含糊不明确
4. Category 原样保持，不影响判断逻辑。
5. 检查点必须一一对应，不得合并、不得跳过。

-------------------------------------------------
【输出格式（极严格要求）】
-------------------------------------------------

必须输出为一个 TSV 代码块，格式如下：

```tsv
Index	Category	Pass
0	FUNCTIONAL	yes
1	BOUNDARY	no
2	EXCEPTION	yes
...
```

要求：

* 使用 TAB 分隔
* Index 从 0 开始递增
* Category 必须与输入检查点保持一致
* Pass 为 yes 或 no
* 不得输出多余的文字、注释、空行、解释说明
* TSV 代码块外不得出现任何其他内容
~~~

# **完整的“基于参考需求的多维可量化评估体系”**

两步流程：

### **A. 从参考需求中抽取“多维检查点”（原子化真值集）**  
→ 这一步确定“真相是什么”。

### **B. 用这些检查点量化评估候选需求文档（yes/no）**  
→ 这一步给出“覆盖率、流程完整性、边界完整性、异常覆盖、冲突一致性”等全部维度的可量化结果。

整个过程是：

- 客观  
- 可重复  
- 可审计  
- 可统计  
- 可用于自动化需求评审 / 流水线 / CI / 回归
